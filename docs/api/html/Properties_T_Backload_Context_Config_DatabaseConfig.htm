Properties
  NameDescription
Connection
Name of the database connection (see Web.config) [Default: "FilesContext"]
ConnectionString
Database connection string (see Web.config)
Extension
Name of the extension file, e.g. extension="Backload.Plugin.Database.dll" [Default: "Backload.Plugin.Database.dll"].
FileTableName
Storage mode SqlFileTable only. Name of the SQL Server FileTable if SqlFileTable storage is used. [Default: "FileTable"]. In a SQL Server FileTable, files will be stored outside of the database file in a special file system directory, but managed by the database engine. This allows to store very large files. In contrast to SQL Server FILESTREAMs, SQL Server FileTable files can be accessed, created and deleted directly within the file system like normal files. This allows you for example to copy a file with the explorer to the special FileTable's directory where SQL Server catches the file and the database engine manages it. When you create a virtual path in IIS to the FileTable directory, a client can download the files directly from the browser. (). FileTables (SQL Server).
MappingTable
Only needed if custom column names are used, e.g. an existing data model. IMappingTable provides a list of column names used to create SQL command strings including original and custom column names. This setting has no effect, if the default column names are used. (If you have custom column names in storage mode "SqlFileStream", the custom column name of the Data column must be configured. Note: You can also set column mapping in your DbContext class if you derive this class from IBackloadMappedStorageProvider instead of IBackloadStorageProvider.
RootPath
Used for storageMode "SqlFileTable" or "FileSystem" only: Root path of the file directory when using external file storage. Note: If rootPath is not configured: In storageMode "FileSystem" default rootPath is set to "~/Files". In storageMode "SqlFileTable" default rootPath is automatically retrieved from SQL Server.
StorageMode
Defines if files are stored within the database file or in the filesystem, e.g. as SQL Server FILESTREAM, as regular file in a SQL Server FileTable or as normal file. [Default: "Internal"] Internal: Files will be stored within the database file (default). SqlFileStream: SQL Server FILESTREAMs are stored outside of the database file as blob within the file system, but they are managed by the database engine. When using SQL Server FILESTREAMs, SQL Server and the database must enable the SQL Server FILESTREAM feature. More info: FILESTREAM (SQL Server). SqlFileTable: Like SQL Server FILESTREAMs, files will be stored outside of the database file in a special file system directory, but managed by the database engine. This allows to store very large files. In contrast to SQL Server FILESTREAMs, SQL Server FileTable files can be accessed, created and deleted directly within the file system like normal files. This allows you for example to copy a file with the explorer to the special FileTable's directory where SQL Server catches the file and the database engine manages it. When you create a virtual path in IIS to the FileTable directory, a client can download the files directly from the browser (). FileTables (SQL Server). FileSystem: Files will be stored on the file system. A path in the database table column points to the file.
TableName
Name of the underlying database table [Default: "Files"].
TempFolder
If set to a none empty string, this setting is used to store file chunks, before the full uploaded file will be stored into the database. If you use chunked file uploads, it is recommended to store the chunks in a temporary directory to get the best performance. An empty string means, that chunk files are stored and merged within the database directly directly [Default: ""].
VirtualDirectory
Storage mode "FileTable" or "FileSystem". This setting is only needed if direct download of files is enabled in the configuration with the setting filesUrlPattern="{url}" (default: {backload}). Note: Direct download of files is only available in the storage modes "SqlFileTable" and "FileSystem", because in these modes files are stored as regular files in the file system. As in standard filesystem storage, VirtualDirectory needs to be set as the name of a virtual directory in IIS or a absolute URL if the file system to download files is outside the web application (e.g. UNC shares). VirtualDirectory represents the name of virtual path in your IIS application to an external storage location. If filesRoot is relative to the web application root (starts with '~/', see filesRoot), you do not need to set this value. If filesRoot is an absolute path, you usually set a virtual path in IIS to this external location. In this case, the virtualDirectory attribute must be set to the name of the virtual path and filesRoot to the absolute path. Example: If the virtual path is z:\files and the name is "files", set filesRoot="z:\files" and virtualDirectory="/files" [Default: "/"]
